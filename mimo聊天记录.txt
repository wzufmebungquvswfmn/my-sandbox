我想知道这个项目有哪些功能
这个项目是个 Rust 命令沙箱，核心是在受控环境中执行外部命令。具体功能：


1. 命令白名单

只允许执行配置过的命令（默认 echo python python3 node cargo ls cat pwd date whoami），rm、curl 之类的会被直接拒绝。


2. 策略控制（TOML 配置文件）

命令白名单 — 哪些命令能跑
目录白名单 — 哪些工作目录允许进入（allowed_roots）
超时上限 — 命令执行时间不能超过这个值（默认 30s）
环境变量过滤 — 只放行指定的 key（默认 PATH HOME USER LANG MODE），其他环境变量会被过滤掉，防泄露

3. Linux 命名空间隔离（Unix only）

toml
toml
[isolation]
enabled = true
rootfs = "/path/to/rootfs"
mount_proc = true
net_namespace = true

CLONE_NEWNS — 挂载命名空间隔离
CLONE_NEWPID — 进程隔离
CLONE_NEWUTS — 主机名隔离
CLONE_NEWIPC — IPC 隔离
CLONE_NEWNET — 网络隔离（可选）
支持 chroot 到指定 rootfs
挂载独立的 /proc

4. Extensions（扩展命令）

在配置文件里预定义命令模板，给一个固定命令加默认参数、环境变量、单独的超时和 env key 白名单。调用时只传额外参数就行，不用每次都指定完整命令。


5. 批量执行

HTTP API 支持一次提交最多 20 个命令并行执行。


6. 审计日志

每次执行都记录：来源、命令、参数、工作目录、结果、耗时
内存保留最近 1000 条
持久化到 JSONL 文件
按 request_id 可查询

7. Metrics 统计

HTTP API 提供执行统计：

总数 / 成功 / 失败 / 拒绝 / 超时
平均耗时、P95、P99

8. 双模式运行

CLI 模式 — 单次执行，结果输出 JSON
HTTP 服务模式 — 起一个 axum 服务，提供 REST API

9. 输出控制

stdout/stderr 各限制 256KB，防止命令输出撑爆内存
超时后自动 kill 子进程


一句话总结： 这是一个面向 AI Agent 或自动化场景的安全执行层——你把命令丢给它，它帮你检查白名单、过滤环境变量、隔离执行、记录审计日志，然后返回结构化结果。

好，我认真看了一遍，说几个真实的不足，不只是表面的"少个 README"，而是面试官可能抓着问的问题：


🐛 有实际 Bug 或逻辑问题

1. allowed_roots 形同虚设policy.rs 里，只有当 cwd 有值才检查目录白名单。如果请求不传 cwd，直接在进程当前目录执行，目录限制完全被绕过。这是一个实际的安全漏洞。


2. percentile 函数有误api.rs 里的 percentile 实现有问题：输入 sorted_data 已经排序了，但函数内部又排序了一遍（多余）。更重要的是，用 round() 做插值近似，在小数据集上 p95 和 p99 结果可能完全一样，甚至反直觉。面试官一问就露馅。


3. API 的 execute 没校验空命令
发一个 {"command": ""} 进去，executor 会尝试 spawn 一个空命令，直接报错但返回 Failed 而不是 Rejected——语义不对。


🔒 安全方面粗糙

4. Linux 隔离用了 namespace 但没做 seccomp
有 unshare 和 chroot，但没有 seccomp filter。被隔离的进程仍然可以调用很多危险的 syscall。对于一个简历上的"沙箱"项目，面试官大概率会问这个。


5. 没有丢弃 capabilitiespre_exec 里没有 capset / prctl(PR_SET_NO_NEW_PRIVS)，namespace 隔离的效果大打折扣。


6. 无鉴权
HTTP API 没有任何 auth，localhost 也挡不住别人调你的接口。


🏗️ 工程方面

7. 错误处理用 String 而非类型PolicyViolation 只有三个 variant，每个都带 String。更规范的做法是给每个错误类型足够信息，方便上游做 i18n 或 structured logging。


8. audit 文件写入没有 flush / fsync
每次写 JSONL 用的是 OpenOptions::append，写完没 flush，进程崩溃最后几条记录可能丢失。


9. AuditStore::list() clone 整个 Vec
高并发下，每次调 /executions 和 /metrics 都会 clone 全部记录（最多 1000 条），然后遍历。没有分页、没有索引、没有读写分离。


10. 测试太薄
只有 7 个测试，全在 happy path 附近。缺少：

Extension 调用测试
批量执行测试
并发安全测试
输出截断测试（256KB 边界）
超时后进程确实被 kill 的验证
API 层集成测试

11. 没有 graceful shutdowntokio::net::TcpListener 直接 serve，Ctrl+C 没有信号处理，正在跑的命令不会被清理。


12. Cargo.toml 都没给
依赖版本完全不知道，不知道能不能编译。
